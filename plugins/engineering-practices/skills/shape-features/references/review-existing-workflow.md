## Context

The user has existing requirements they want evaluated. These may be:
- User stories in a backlog
- A product requirements document (PRD)
- Feature specifications
- Meeting notes that serve as de facto requirements
- AI-generated requirements from a product brief or conversation
- Jira/Linear/GitHub issue descriptions

The review applies the same quality bar regardless of who authored the requirements (human or AI) or what format they're in. The goal is to produce clear, actionable feedback that helps improve them.

## Process

**Pacing:** follow the Conversation Style rules from SKILL.md throughout. Each step's *Why this step matters* note is user-facing - share it briefly as the step begins. For inputs larger than about five stories, review in chunks: present findings for one section, check in, continue.

## Step 1: Receive the requirements

*Why this step matters: a review with no baseline context is guesswork. Knowing what the user has, who wrote it, and how old it is determines whether the review adds value or just produces noise.*

Ask the user to provide their requirements. Accept any format - the quality check applies to the content, not the format.

If the user mentions that the requirements were generated by an AI agent, flag this for Step 3 (AI-specific checks apply).

**Triage gate - is this actually a review job?**

Before running the quality check, classify what the user provided:

| Input looks like… | Action |
|-------------------|--------|
| Structured requirements - stories, acceptance criteria, PRD sections, feature specs, issue descriptions with clear form | Proceed to Step 2. |
| Raw input - meeting notes, chat log, bullet-point brainstorm, a single paragraph description, a loose feature idea | Offer to switch to [shape-new-feature-workflow.md](shape-new-feature-workflow.md), which is designed to turn raw input into structured requirements. If the user wants to stay in review, proceed - but state explicitly that every finding will be structural ("no acceptance criteria", "no value statement", "no scope boundary") rather than content-level, and that the more useful next step is usually to shape a new feature rather than review. |
| Mixed - some structured stories alongside loose notes | Proceed with review, but scope the review to the structured items and treat the loose notes as open follow-ups in the final verdict. |

Running the full quality checklist against unstructured input produces a wall of "missing" findings that are not actionable. The triage keeps the review focused on input it can actually improve.

**Check for staleness.** Ask when these requirements were written or last refined. If they are more than a few weeks old, flag this prominently: stale requirements are one of the most common sources of wasted work. The product may have evolved, assumptions may no longer hold, dependencies may have changed, and priorities may have shifted. Stale requirements should be treated as input for a new refinement conversation, not as ready-to-build specifications. This is a Critical finding if the user intends to start work from these requirements without re-validating them.

## Step 2: Structured quality check

*Why this step matters: a quality check with a fixed rubric produces findings that are consistent, comparable, and defensible. Ad-hoc reviews surface whatever the reviewer happens to notice on the day.*

Evaluate each requirement (or story, or feature description) against the quality gate. For EACH item, check:

**A. Value clarity**
- Is there a clear statement of WHY this work matters?
- Can you identify who benefits and what problem is solved?
- If the value is missing or vague, flag it.

**B. Testable acceptance criteria**
- Are there concrete Given/When/Then scenarios?
- If criteria exist but aren't in scenario format, are they at least specific and verifiable?
- If there are no acceptance criteria at all, this is a critical finding.

**C. Declarative vs. imperative**
- Do the criteria describe outcomes (what) or implementation steps (how)?
- Are UI interactions prescribed where behaviors should be described?
- Flag any criteria that embed implementation decisions.

**D. Size and deliverability**
- Could this story be delivered to production in under 2 days?
- If it feels larger, identify where it could be decomposed.
- Flag stories that are clearly epics masquerading as stories.

**E. Vertical slicing**
- Does each story deliver an observable, end-to-end behavior?
- Are there stories that only touch one layer (only DB, only API, only UI)?
- Are there stories that must be deployed together to produce any value?
- Are there stories sliced by activity (coding, testing, review)?

**F. Edge cases and unhappy paths**
- Are there scenarios for invalid inputs, failures, and unexpected conditions?
- Is only the happy path covered?
- Are the unhappy paths genuine or formulaic (just inverted happy paths)?

**G. Non-functional requirements**
- Are performance, security, accessibility, observability, and data handling addressed?
- Are non-functional criteria specific ("under 500ms") or vague ("should be fast")?

**H. Unknowns and assumptions**
- Are assumptions called out explicitly, or presented as facts?
- Are there unknowns that should be spikes?
- Are there dependencies that aren't acknowledged?

**I. Multi-audience readability**
- Could a product owner confirm this matches their intent?
- Could a developer start building from this without asking questions?
- Could a tester write tests from this right now?
- Could an AI agent produce correct code from these criteria?

## Step 3: Authorship-specific checks

*Why this step matters: AI and humans fail in different ways, and both kinds of failure look plausible on the page. AI backfills unknowns with invented defaults; humans carry forward whatever they were given without testing it. The review must catch both - not just the fashionable one.*

**Run the AI-authorship checks if the requirements were AI-generated** (stated by user or evident from the writing style):

- **Formulaic edge cases:** AI-generated unhappy paths often mirror the happy path structure rather than reflecting genuine failure modes. Check: do the edge cases feel like real scenarios someone would encounter, or do they feel like mechanical inversions?
- **Plausible-sounding vagueness:** AI output tends to use the right vocabulary while remaining imprecise. Check: are the acceptance criteria actually specific, or do they sound specific without committing to concrete values?
- **Assumption backfill:** AI agents fill gaps with reasonable-sounding assumptions. Check: are there statements presented as requirements that are actually unvalidated assumptions? (e.g., "The system supports up to 10,000 concurrent users" - says who?)
- **Symmetrical completeness:** AI-generated specs tend to cover all the "right" sections without depth. Check: are non-functional requirements genuinely thought through, or are they generic boilerplate?
- **Missing "I don't know":** AI rarely says "this needs investigation." Check: are there areas where uncertainty SHOULD exist but isn't expressed?

**Always run the human-authorship checks** - these apply regardless of stated authorship, because a human reviewer will have been involved somewhere and their failure modes propagate:

- **Stakeholder echo:** the value statement repeats a stakeholder's verbatim phrasing without translation into observable behavior. Check: are the author's words and the stakeholder's words distinguishable?
- **Seniority-as-evidence:** claims of the form "per the architect" or "per the tech lead" with no reasoning attached. Check: can the claim be verified without asking the cited person?
- **Sneaked implementation:** URL paths, table names, widget types, library names, or specific UI text in the acceptance criteria. Check: if you replace a noun with a synonym, does the meaning change? If yes, it is imperative, not declarative.
- **False completeness:** multiple scenarios that are the same scenario with different nouns. Check: what behavior does each scenario verify that the others do not? If the answer is just "the filter value is different," they are one scenario.
- **Anchoring on the first idea:** every story descends from the first framing; no scenario challenges the original stakeholder's premise. Check: look for negative space - what was never asked?
- **Absent user:** the feature is described from the system's perspective; every Then is a UI state or data change, with no human interpretation or judgment. Check: does any scenario reflect what a real person would *think* when this happens?

Read the `<ai-authorship-examples>` and `<human-authorship-examples>` sections in [review-examples.md](review-examples.md) for annotated examples of each failure mode in real requirements.

## Step 4: Produce findings

*Why this step matters: vague feedback ("needs improvement") does not change behavior. Findings that name what is wrong, why it matters, and what to do instead are the only form of review that actually moves a requirement toward buildable.*

For each finding, provide:

1. **What:** The specific issue found (quote the problematic text)
2. **Why it matters:** The real-world consequence if this isn't fixed (not just "best practice says so" - what actually goes wrong)
3. **What to do instead:** A concrete rewrite or specific guidance for improvement

Organize findings by severity:

- **Critical:** No acceptance criteria, no value statement, story is clearly an epic, or requirements are untestable. Work should NOT start until these are addressed.
- **Major:** Missing edge cases, vague non-functional requirements, horizontal slicing, hidden assumptions. Work can start but will likely hit problems.
- **Minor:** Imperative phrasing, formatting issues, criteria that could be slightly more specific. Improve if time permits.

Read the `<sample-findings>` section in [review-examples.md](review-examples.md) for a worked example showing findings at each severity level with concrete rewrites.

**Volume control - findings must inform, not bury:**

- **Lead with the verdict.** Present the Step 6 summary (counts + Definition of Ready call) *before* the detailed findings, so the user gets the decision first and the evidence second.
- **Group repeated findings.** "8 of 12 stories have no NFRs" is one finding with a list of affected stories, not eight findings. A grouped finding teaches the pattern; eight copies bury it.
- **Top findings in full, the rest on request.** Present the 3-5 most consequential findings with the full what / why / what-instead structure. List the remainder as one line each and offer detail on request.
- **Long reviews go to a file.** If the full findings list is long, write it to a file (offer `docs/reviews/<name>.md` or alongside the source document) and keep chat to the verdict, the top findings, and next steps.

## Step 5: Provide a rewritten version

*Why this step matters: showing what "good" looks like for the user's specific requirement is worth more than any amount of general advice. It converts the review from a grading exercise into a collaborative improvement.*

For requirements with Critical or Major findings, provide one or two **exemplar rewrites** - pick the findings whose correction teaches the most transferable pattern. Showing the pattern once teaches more than rewriting everything; an exhaustive rewrite reads as grading, an exemplar reads as teaching. Then offer to rewrite the remainder on request, or to apply the rewrites directly to the source file or document the user provided.

For each rewrite:
- Add value statements where missing
- Convert imperative criteria to declarative
- Add edge case scenarios
- Surface hidden assumptions
- Suggest vertical slicing where stories are too large

## Step 6: Summary verdict

*Why this step matters: the verdict converts a list of findings into a decision - start work, rework first, or go back to discovery. Without it the user is left with a pile of issues and no direction.*

Per the volume-control rule in Step 4, the verdict is *presented* before the detailed findings even though it is *produced* after the evaluation. Use this step to confirm or revise it once rewrites are done - do not repeat it verbatim at the end.

Provide an overall assessment:

- **Total items reviewed:** [count]
- **Critical findings:** [count] - these must be fixed before work starts
- **Major findings:** [count] - these will cause problems during delivery
- **Minor findings:** [count] - improvements that increase clarity

State clearly whether the requirements meet the Definition of Ready as a whole. If they don't, identify the minimum changes needed to get there.

## Step 7: Close the loop - suggest next steps

*Why this step matters: a review that ends at the verdict tells the user what's wrong but not what to do about it. Closing the loop is the difference between feedback and an improvement.*

Based on the verdict, propose concrete next steps:

- **Critical findings present.** Recommend reworking via the `shape-new-feature` workflow before any build work starts. For AI-authored requirements with critical findings, make the human-review stance explicit: the draft is a starting point; the skill's goal is to help the human reviewer fix it before it becomes buildable.
- **Major findings only.** Offer to help revise the most impactful items inline, or to re-run the `shape-new-feature` workflow scoped to the affected stories.
- **Minor findings only / clean bill of health.** Point toward the downstream activities the requirements enable - system or architecture design for decisions the requirements imply, a testing-strategy pass to plan verification, code review once implementation begins. If the user has dedicated skills for any of those in their environment, name them; otherwise describe the activity. The requirements are input to that next step, not a substitute.
- **Stale requirements.** Regardless of finding severity, if staleness was a Critical flag in Step 1, the primary next step is a fresh discovery conversation - not a patch to the existing document.
- **Missing stakeholders.** If the multi-audience check reveals that one of PO/dev/tester was not involved in the original conversation, schedule that conversation before any work proceeds. The scenarios are only valuable if the right people have actually read and agreed to them.

Keep the closer specific: name the skill or activity, don't hand-wave.

## Success Criteria

This workflow is complete when:
- [ ] All provided requirements have been evaluated against the quality gate
- [ ] Authorship-specific checks applied: AI checks when AI-authored, human checks always
- [ ] Each finding includes what's wrong, why it matters, and what to do instead
- [ ] Findings are organized by severity (critical/major/minor)
- [ ] Exemplar rewrites provided for the most instructive critical/major findings, remainder offered on request
- [ ] Overall verdict states whether requirements meet Definition of Ready
- [ ] User has received actionable, specific feedback (not vague "needs improvement")
